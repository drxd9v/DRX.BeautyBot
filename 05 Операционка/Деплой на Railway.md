# Деплой на Railway

> Навигация: [[Индекс Obsidian]] · [[Навигация по операционке]]
>
> Связанные документы: [[Стратегия хостинга]] · [[Схема хостинга на старте]] · [[Чеклист запуска]]

## Для чего это подходит
Railway подходит, если нужно:
- быстро поднять demo;
- не возиться с VPS на старте;
- держать бота как always-on сервис;
- сохранить SQLite через volume.

## Что уже подготовлено в проекте
В проекте уже добавлена поддержка переменной:
- `DB_PATH`

Это позволяет хранить SQLite не в контейнере, а на volume Railway.

## Как деплоить

### 1. Загрузить проект на GitHub
Railway удобнее всего подключать через GitHub-репозиторий.

Важно:
- не пушить живой `.env` с токенами;
- использовать `.env.example` как шаблон переменных.

### 2. Создать проект в Railway
В Railway:
1. Нажать `New Project`
2. Выбрать `Deploy from GitHub repo`
3. Подключить нужный репозиторий

### 3. Добавить переменные окружения
В `Variables` добавь как минимум:
- `BOT_TOKEN`
- `ADMIN_ID`
- `DEVELOPER_ID`
- `OWNER_ID`
- `MANAGER_IDS`
- `PAYMENT_PROVIDER_TOKEN`
- `PAYMENT_CURRENCY=USD`
- `MINI_APP_URL`
- `DB_PATH=/data/bot.db`

Опционально:
- `STAFF_ID`
- demo video URL переменные, если они реально используются

### 4. Подключить volume
Так как бот использует SQLite, нужен volume.

Сделай так:
1. Создай `Volume`
2. Подключи его к сервису
3. Укажи mount path: `/data`

После этого база будет жить в:
- `/data/bot.db`

### 5. Указать Start Command
В `Start Command` задай:

```bash
python main.py
```

Даже если Railway попытается определить команду сам, для этого проекта лучше зафиксировать её явно.

### 6. Задеплоить
После этого:
1. Проверь staged changes
2. Нажми Deploy
3. Дождись старта сервиса
4. Посмотри логи

## Что проверить после деплоя
- бот стартовал без ошибок;
- нет ошибки по `BOT_TOKEN`;
- нет ошибки по `ADMIN_ID`;
- база создаётся в `/data/bot.db`;
- бот отвечает на `/start`;
- после рестарта данные не пропадают.

## Важные замечания

### Не храни SQLite без volume
Если не подключить volume, база будет жить в файловой системе контейнера и может потеряться при redeploy.

### Не используй pre-deploy для работы с SQLite-файлом
Volumes в Railway не монтируются в pre-deploy фазе.
Поэтому всё, что связано с локальной SQLite-базой, должно происходить во время старта сервиса.

### Railway подходит прежде всего для demo
Для первых запусков Railway удобен.
Но если потом захочется минимальной себестоимости и более плотного контроля, можно переносить клиентов на VPS.

## Моя рекомендация

### Для demo
Railway — нормальный вариант, если хочется быстро стартовать без лишней инфраструктуры.

### Для клиента
Можно тоже Railway, если важнее простота.
Если важнее цена и контроль — потом переносить на VPS.
